  目的:
  既存ソフトへ UI編集拡張を最小差分で移植する。既存挙動を壊さないことを最優先とする。

  実装方針:
  - src/tools/ui_edit/ を機能単位で追加する
  - 既存側の変更は入口接続のみ（原則1〜3ファイル以内）
  - 既存コードからは tools::ui_edit::api のみ参照し、内部実装へ直接依存しない
  - 既存のコマンドIDとUI定義形式の互換性を維持する
  - 無関係なリファクタリングは禁止

  前提:
  - 初回起動でUI位置やサイズが初期配置になる可能性がある
  - runtime/ui/live/ui.json を事前配置する（未配置だと起動時読込で失敗し得る）

  変更許可:
  - src/tools/**
  - 入口接続に必要な既存最小ファイルのみ

  変更禁止:
  - 実装範囲外の既存ロジック変更
  - 既存コマンド仕様の破壊
  - 依存クレートの不要追加

  依存クレート要件（移植先プロジェクト）:
  - 実装開始前に移植先 `Cargo.toml` を確認する
  - 以下を必要クレートとし、未導入が1つでもあれば実装を開始しない
    - eframe（egui を含む。UI編集画面の描画とViewport制御に必須）
    - serde（derive を有効化。UI定義のシリアライズ/デシリアライズに必須）
    - serde_json（runtime/ui/live/ui.json の読込/保存に必須）
  - 推奨（既存実装が Result + Context 方式の場合のみ）
    - anyhow
  - 上記以外のGUIクレート追加（例: iced / gtk / slint）は禁止
  - バージョンは移植先プロジェクトの既存バージョンを優先し、互換を崩す更新は行わない

  実装中止条件:
  - 必要クレート（eframe / serde(derive) / serde_json）が未導入
  - 既存依存との競合で同時利用が困難
  - バージョン制約により互換性を満たせない
  - 上記に該当した場合は実装を開始せず、不足クレート名と中止理由を記録して終了する

  受け入れ条件:
  1. debug build 成功
  2. 拡張OFF時の既存挙動が維持される
  3. 拡張ON時にUI編集画面が表示できる
  4. 編集→保存→再起動で反映が維持される

  失敗時:
  - 入口接続差分をロールバックして既存経路へ戻す
  - 失敗内容は再現手順付きで記録する
